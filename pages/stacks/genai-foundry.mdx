The **GenAI Foundry** stack deploys an [Azure AI Foundry](https://learn.microsoft.com/en-us/azure/foundry/) account as the central infrastructure layer for your GenAI workloads. It provisions all dependent resources (storage, Cosmos DB, AI Search, Key Vault) with managed identity authentication and supports private networking with VNet injection for Agent APIs.

<Warning>
**Deploy and destroy operations can take up to 180 minutes** when using private networking (`selected_networks` or `disabled`). This is caused by Azure's internal provisioning of VNet injection infrastructure for the Agent Service — Azure provisions a managed Container Apps environment, configures network peering, and sets up DNS forwarding behind the scenes. These Azure-side operations are not visible in Terraform output but run asynchronously during the `networkInjections` update and capability host creation. Plan CI/CD pipeline timeouts accordingly.
</Warning>

## Architecture

The stack creates a single resource group containing:

| Resource | Required | Purpose |
|---|---|---|
| AI Foundry Account | Yes | Central account for AI model management and agent hosting |
| Storage Account | Yes | Agent file storage (shared access keys disabled) |
| Cosmos DB | Yes | Agent thread storage (local auth disabled) |
| AI Search | Yes | Vector store for knowledge bases and RAG |
| Key Vault | Yes | Secrets and key management |
| Log Analytics Workspace | Yes | Centralized logging |
| Application Insights | Yes | Monitoring and telemetry |
| Virtual Network | Conditional | Auto-created when using private networking |
| Private Endpoints | Conditional | Secure access to all resources |
| Account Capability Host | Conditional | Enables Agent Service with VNet injection |

### Deployment strategy

The Foundry account is deployed in two phases to avoid Azure provisioning timeouts:

1. **Phase 1** -- Create the Foundry account without VNet injection (completes in minutes)
2. **Phase 2** -- Update the account to add `networkInjections` for Agent APIs
3. **Phase 3** -- Create the account-level capability host referencing the agent subnet

This pattern matches the Azure portal's behavior and avoids the known timeout issues with single-step VNet injection deployments.

## Networking

The stack supports three network security postures, matching the Azure portal's inbound access options.

<Tabs>
<Tab title="Public">
  All resources are accessible over the public internet. No VNet, private endpoints, or VNet injection are created.

  Best for quick prototyping and development without compliance constraints.
</Tab>

<Tab title="Selected Networks">
  Public access is restricted to specified IP ranges. Private endpoints and VNet injection are deployed.

  - Storage, Cosmos DB, AI Search, and Key Vault default to **Deny** with firewall IP rules
  - VNet injection enables Agent APIs through a delegated subnet
  - Private DNS zones are auto-created or you can supply existing ones
</Tab>

<Tab title="Disabled">
  The most restrictive mode. **No public network access** for any resource.

  - All traffic flows exclusively through private endpoints
  - VNet injection with a dedicated agent subnet (delegated to `Microsoft.App/environments`)
  - The Foundry account uses private endpoints for inbound access
  - Agents operate within the customer VNet for full network isolation
</Tab>
</Tabs>

### Private endpoints

When using a private networking mode, private endpoints are created for:

| Resource | Sub-resource |
|---|---|
| Storage Account | Blob |
| Key Vault | Vault |
| Cosmos DB | SQL |
| AI Search | Search Service |
| Foundry Account | Account (3 DNS zones) |

<Tip>
You can bring your own VNet, subnets, and DNS zones or let the stack auto-create them. The agent subnet requires a minimum size of /27 (recommended /24) and must be delegated to `Microsoft.App/environments`.
</Tip>

### VNet injection for agents

When `network_security` is set to `selected_networks` or `disabled`, the stack configures VNet injection:

- A dedicated subnet delegated to `Microsoft.App/environments`
- The Foundry account is updated with `networkInjections` referencing the agent subnet
- An account-level capability host binds the Foundry to the subnet
- Agent runtime traffic flows through the customer VNet with full DNS and routing control

## Security

The stack enforces managed identity authentication wherever possible:

| Resource | Authentication |
|---|---|
| Storage Account | Managed Identity (shared access keys **disabled**) |
| Cosmos DB | Managed Identity (local authentication **disabled**) |
| AI Search | AAD + API Key (configurable) |
| Key Vault | RBAC-based access |

The Foundry account's system-assigned managed identity is granted:

| Target Resource | Roles |
|---|---|
| Key Vault | Key Vault Secrets User |
| Storage Account | Storage Blob Data Contributor |
| Cosmos DB | Cosmos DB Operator, DocumentDB Account Contributor, SQL Data Contributor (data plane) |
| AI Search | Search Index Data Contributor, Search Service Contributor |
